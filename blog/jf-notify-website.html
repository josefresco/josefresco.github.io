Building jfnotify.com: Designing a Static Marketing Site for a WordPress Plugin

Web Development March 2026
TL;DR: Once JF Notify was ready to ship, it needed a home that wasn't a GitHub README. jfnotify.com is a purpose-built static HTML/CSS marketing site hosted on GitHub Pages — no framework, no build step, no CMS. This post covers the design decisions, landing page structure, SoftwareApplication schema markup, and why static HTML is often the right choice for a single-product plugin site.

A Plugin Needs a Landing Page

When JF Notify was just a GitHub repository, the only way to understand what it did was to read the README. That's fine for developers browsing GitHub, but it's not where WordPress site owners end up when they're looking for a solution to a problem.

The plugin needed a dedicated URL — something I could link to from the WordPress plugin directory, from blog posts, and from my portfolio. A landing page that answers the visitor's core question in the first ten seconds: "Will this solve my problem, and how do I get it?"

The choice was between building something with WordPress (the obvious choice for a WordPress plugin's marketing site), a static site generator like Eleventy or Hugo, or just hand-coded HTML. I went with hand-coded HTML for the same reason I tend to reach for it for single-purpose sites: no dependencies to manage, no build pipeline to maintain, no CMS to update. The site is three files — index.html, style.css, and a handful of assets. It deploys by pushing to GitHub.

What the Page Needs to Do

Before writing a line of HTML, I thought through what a visitor arriving at jfnotify.com actually needs to accomplish:

  • Understand the product in ten seconds — What is JF Notify? What problem does it solve? Who is it for?
  • Trust that it works — Enough technical detail and specificity that a developer or agency owner feels confident installing it on a client site.
  • Find the download or installation path — The primary CTA is getting the plugin. That path needs to be obvious and frictionless.
  • Understand the integration requirements — JF Notify requires both Gravity Forms and a Telegram bot. Visitors need to know that upfront, not discover it after installation.

Those four goals directly shaped the page structure. Each section of the page maps to one of them.

Page Structure: The Anatomy of a Plugin Landing Page

The page is organized into six distinct sections, each with a single job:

1. Hero — The Ten-Second Test

The hero section leads with a single sentence that names the product, states what it does, and names the tools it connects: "Instant Telegram notifications for Gravity Forms submissions." No jargon, no passive voice, no "leveraging synergies." Then a short supporting paragraph with slightly more context, and two CTA buttons — one primary (get the plugin) and one secondary (view on GitHub for developers who want to inspect before installing).

The visual element in the hero is a simplified diagram showing the flow: Gravity Forms submission → JF Notify → Telegram notification. Diagrams answer "how does this work?" faster than paragraphs do.

2. Feature Cards — Trust Through Specificity

Generic feature lists ("Easy to use!", "Powerful!", "Reliable!") do nothing for trust. The feature cards on jfnotify.com are specific: "Field mapping — choose exactly which form fields appear in your Telegram message", "Conditional logic — only notify when specific form values are submitted", "Retry on failure — failed deliveries are retried automatically with exponential backoff."

Specificity is the credibility signal. When a developer reads "exponential backoff," they know the plugin was built by someone who thought about edge cases. That detail does more conversion work than any marketing adjective.

3. Requirements — Set Expectations Early

This section gets skipped on a lot of plugin landing pages, and it's a mistake. JF Notify has two hard requirements: an active Gravity Forms license and a Telegram bot token. If a visitor doesn't have those, installing JF Notify won't help them — and they'll leave a bad review blaming the plugin for something that was a compatibility issue.

A short "What You'll Need" section before the install instructions saves support emails and sets honest expectations. It also filters in the right audience: WordPress shops that already use Gravity Forms are the target users, and naming Gravity Forms prominently helps those visitors self-identify.

4. Installation — From Download to First Notification

The installation section walks through the process step by step: download the plugin, install via WordPress admin or WP-CLI, activate, navigate to the settings page, enter the Telegram bot token and chat ID, map the form fields. Each step is concrete and numbered.

The goal is to get a new user to their first working notification as fast as possible. First value achieved = product credibility established. I included a screenshot of the settings page here because nothing short-circuits setup confusion like a visual reference.

5. FAQ — Defuse the Objections

The FAQ section addresses the questions that show up repeatedly in support: "Does this work with Gravity Forms conditional logic?", "Can I send to multiple Telegram chats?", "What happens if Telegram is down when a form is submitted?", "Is the bot token stored securely?"

These aren't hypothetical questions — they're the real things people wonder about before installing a notification plugin on a client site. Answering them on the page reduces the friction between "looks interesting" and "I'll try it."

6. Footer CTA — Close with Action

The page ends where it started: a clear CTA to get the plugin. Visitors who scrolled to the bottom read everything and are the most qualified to convert. Meeting them again at the bottom with a direct link is good conversion practice.

Design Direction: Matching the Ecosystem

JF Notify is part of the JF product family, so jfnotify.com shares the same dark-theme design language as this portfolio site — the same neural-network-inspired color palette (cyan, purple, amber), the same card patterns, the same Inter typeface. Consistency across the product family makes each property feel like it belongs to the same developer.

For a plugin targeting WordPress agencies and developers, the dark developer aesthetic works well. The audience is technical, comfortable with dark themes, and more impressed by clean code samples in the install instructions than by hero images of people at laptops.

The Telegram brand color (#27A7E7) is used as a secondary accent in the integration-focused sections — a small touch that signals "this product was built by someone who actually uses Telegram."

Hosting: GitHub Pages + Custom Domain

The jf-notify-website repository is a public GitHub repository with a CNAME file pointing to jfnotify.com. GitHub Pages serves it over HTTPS with automatic certificate renewal. The setup takes about five minutes and costs nothing beyond the domain registration.

The deployment workflow is as simple as it gets:

# Make changes locally
# Commit and push
git add .
git commit -m "Update feature descriptions"
git push origin main
# GitHub Pages serves the update within ~30 seconds

No build step. No CI/CD pipeline. No deployment config. For a static marketing site that changes infrequently, this is exactly the right level of complexity.

One thing worth noting: GitHub Pages serves from the repository root by default. The entire site lives in index.html at the root, with assets in an /assets subdirectory. This keeps the repository structure obvious and the deployment behavior predictable.

Schema Markup: SoftwareApplication

Most plugin landing pages use generic WebPage or WebSite schema. For a software product page, SoftwareApplication schema is more precise and gives search engines structured information about what the product actually is:

{
  "@context": "https://schema.org",
  "@type": "SoftwareApplication",
  "name": "JF Notify",
  "description": "WordPress plugin for instant Telegram notifications on Gravity Forms submissions",
  "applicationCategory": "BusinessApplication",
  "operatingSystem": "WordPress",
  "offers": {
    "@type": "Offer",
    "price": "0",
    "priceCurrency": "USD"
  },
  "url": "https://jfnotify.com",
  "downloadUrl": "https://github.com/josefresco/jf-notify",
  "softwareRequirements": "WordPress 5.6+, Gravity Forms, PHP 7.4+",
  "author": {
    "@type": "Person",
    "name": "Jose Fresco",
    "url": "https://josefresco.com"
  }
}

The operatingSystem: "WordPress" field is the key detail. Schema.org's SoftwareApplication type was designed for apps but maps cleanly to WordPress plugins — WordPress is the platform the software runs on, the same way iOS is the platform a mobile app runs on.

The offers block with price: "0" signals to search engines that the plugin is free. This can surface in rich snippets that display pricing information alongside the search result.

Performance: No JavaScript Required

jfnotify.com loads with zero JavaScript — no analytics, no chat widgets, no cookie consent banners. The page achieves its conversion goals with HTML and CSS alone.

This is a deliberate choice. JavaScript has real costs on marketing sites: parse time on mobile, third-party script dependencies that can fail, cookie consent requirements that interrupt the first interaction. For a static landing page, eliminating JS entirely removes all of those concerns and makes the page load predictably fast on any device.

The one place JavaScript would add value is analytics — knowing where visitors come from and which sections they engage with. I'm handling that by tracking referrals from the GitHub repository and the blog posts rather than embedding a tracking pixel on the landing page itself. Not perfect, but it's enough signal to make iteration decisions without adding script overhead to the page.

What I'd Do Differently

Looking back at the site after a few weeks of it being live, two things stand out as improvements worth making:

A short demo video in the hero. The installation section does a good job of explaining setup, but a 30-second screen recording showing a Gravity Forms submission arriving as a Telegram notification would be more convincing than any written description. Video is the most effective format for "here's what this does," and it's the one thing the current site is missing.

An explicit "works with" section for common Gravity Forms setups. JF Notify works with Gravity Forms conditional logic, multi-page forms, file uploads, and payment integrations — but the current page doesn't call that out explicitly. Visitors who use those features specifically might not realize they're covered, which means some qualified prospects bounce when they shouldn't.

Both of those are on the roadmap for the next revision of the site.

The Static Site Is the Right Tool

Every few months there's a new argument for why developer-built products should use Webflow, or Framer, or a headless WordPress CMS for their marketing pages. Those tools have their place. But for a single-product plugin site that changes four times a year, hand-coded HTML hosted on GitHub Pages is hard to beat: zero hosting costs, zero maintenance burden, sub-second load times, and a deployment workflow that's literally git push.

The instinct to reach for a more sophisticated tool often reflects a desire to feel productive rather than a genuine need. For jfnotify.com, the right amount of technology is exactly what it has.

Conclusion

jfnotify.com started as a README that needed to become a real product page. The result is a static HTML/CSS site that answers the four questions every plugin landing page visitor has, built with no framework overhead and deployed on GitHub Pages. The design matches the JF product family, the schema markup tells search engines it's a software product, and the whole thing updates with a git push.

The repository is public — if you're building a similar landing page for a plugin or small tool, the structure is worth a look.
