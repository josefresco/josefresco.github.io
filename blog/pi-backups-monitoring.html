Pi Backups: Healthcheck Alerts, Retention Pruning, and Automated Recovery Drills

Raspberry Pi April 2026
TL;DR: The April 2026 pi-backups update adds three companion scripts: healthcheck.sh runs at 3 AM via cron and fires a Telegram alert if no backup archive has landed on the remote in the past 24 hours (a dead-man switch for silent failures); retention.sh runs weekly and deletes GPG-encrypted archives older than a configurable number of days; and recover.sh is an automated restore drill that decrypts and unpacks the latest archive into a staging directory, reports file counts and total size, then removes the staging copy. Together they close the gap between "backups are running" and "backups are working."

The Gap the Original System Left Open

The original pi-backups post covered the nightly rsync workflow that gets Pi application data off the SD card and onto a remote server. The follow-up post added GPG encryption so the archives are safe even if the remote server is compromised. But both posts assumed the backup was actually running.

In practice, backups fail silently all the time:

  • SSH key rotated on the remote, breaking passwordless auth
  • Remote disk full — rsync exits cleanly but writes nothing
  • Cron entry accidentally deleted during a system config cleanup
  • Pi rebooted at 2 AM exactly (cron jobs don't reschedule missed runs by default)

You don't find out about any of these until the moment you need a restore. The three scripts in this update are designed to surface failures before that moment, not after.

healthcheck.sh — The Dead-Man Switch

A dead-man switch is a mechanism that fires when something expected stops happening. The healthcheck script doesn't monitor the backup process — it checks for its output. Once per day, at 3 AM, it SSHes to the remote server and checks the modification timestamp of the most recent archive. If that timestamp is more than 24 hours old, it fires a Telegram alert.

#!/bin/bash
# healthcheck.sh — alert if no backup has landed in the past 24 hours

REMOTE_USER="backup"
REMOTE_HOST="your-remote-server.com"
REMOTE_BACKUP_PATH="/backups/pi"
MAX_AGE_HOURS=24

BOT_TOKEN="${TELEGRAM_BOT_TOKEN}"
CHAT_ID="${TELEGRAM_CHAT_ID}"

send_telegram() {
    curl -s -X POST \
        "https://api.telegram.org/bot${BOT_TOKEN}/sendMessage" \
        -d "chat_id=${CHAT_ID}" \
        --data-urlencode "text=$1" \
        > /dev/null
}

# Find the most recently modified .gpg archive on the remote
LATEST_MTIME=$(ssh "${REMOTE_USER}@${REMOTE_HOST}" \
    "find ${REMOTE_BACKUP_PATH} -name '*.gpg' -printf '%T@\n' 2>/dev/null | sort -n | tail -1")

if [ -z "$LATEST_MTIME" ]; then
    send_telegram "⚠️ Pi backup healthcheck: no archives found at ${REMOTE_BACKUP_PATH}"
    exit 1
fi

NOW=$(date +%s)
AGE_SECONDS=$(( NOW - ${LATEST_MTIME%.*} ))
AGE_HOURS=$(( AGE_SECONDS / 3600 ))

if [ "$AGE_HOURS" -gt "$MAX_AGE_HOURS" ]; then
    send_telegram "⚠️ Pi backup overdue: last archive is ${AGE_HOURS}h old (threshold: ${MAX_AGE_HOURS}h). Check cron and SSH auth."
    exit 1
fi

# Backup is current — no alert needed (silent success)

A few design decisions worth explaining:

  • Silent success. The script sends no message when the backup is current. A daily "everything is fine" notification would train you to ignore it, which defeats the purpose. Alerts should mean something is wrong.
  • It checks the output, not the process. Monitoring whether cron ran backup.sh is less useful than monitoring whether a fresh archive actually exists. The backup script could run and fail in ways cron monitoring wouldn't catch.
  • Credentials from environment variables. TELEGRAM_BOT_TOKEN and TELEGRAM_CHAT_ID are loaded from the environment, not hardcoded. On Raspberry Pi, source them from ~/.bashrc or a dedicated ~/.env file that cron loads via the BASH_ENV variable.

Loading Credentials for Cron

Cron doesn't source ~/.bashrc. To pass environment variables to the healthcheck script, create a credentials file:

# ~/.pi-backups-env
export TELEGRAM_BOT_TOKEN="your-bot-token"
export TELEGRAM_CHAT_ID="your-chat-id"

Then source it in the cron entry:

# /etc/crontab
0 3 * * * pi bash -c 'source /home/pi/.pi-backups-env && /home/pi/scripts/healthcheck.sh'

This keeps credentials out of the crontab file itself (which is world-readable on many systems) and centralizes them in one place that all backup scripts can share.

retention.sh — Pruning Old Archives

The backup script creates a new date-stamped archive every night. Without a retention policy, those archives accumulate indefinitely. retention.sh runs weekly — Sunday morning, after the nightly backup has had time to complete — and removes archives older than a configurable threshold.

#!/bin/bash
# retention.sh — remove backup archives older than KEEP_DAYS days

REMOTE_USER="backup"
REMOTE_HOST="your-remote-server.com"
REMOTE_BACKUP_PATH="/backups/pi"
KEEP_DAYS=30

LOG_FILE="/var/log/pi-backups.log"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] [retention] $1" | tee -a "$LOG_FILE"
}

log "Running retention policy (keep last ${KEEP_DAYS} days)..."

# Count archives before pruning
BEFORE=$(ssh "${REMOTE_USER}@${REMOTE_HOST}" \
    "find ${REMOTE_BACKUP_PATH} -name '*.gpg' | wc -l")

# Delete archives older than KEEP_DAYS
DELETED=$(ssh "${REMOTE_USER}@${REMOTE_HOST}" \
    "find ${REMOTE_BACKUP_PATH} -name '*.gpg' -mtime +${KEEP_DAYS} -print -delete | wc -l")

# Count archives after pruning
AFTER=$(ssh "${REMOTE_USER}@${REMOTE_HOST}" \
    "find ${REMOTE_BACKUP_PATH} -name '*.gpg' | wc -l")

log "Retention complete: ${BEFORE} archives before, ${DELETED} removed, ${AFTER} remaining."

The script logs counts before and after so you can audit what was removed. If you see the archive count drop unexpectedly, the log entry gives you the numbers to investigate.

Thirty days covers the most common recovery scenario — "I deleted something important last week" — without consuming unlimited remote disk space. The KEEP_DAYS variable is at the top of the script so you can tune it per deployment without hunting through the code.

Why Not Use find's -delete on the Remote Directly?

The -print -delete combination in find prints each deleted path before removing it, which gives us the count. Running this via SSH means the deletion happens on the remote server without transferring any file data over the network — only the command is sent, not the archives themselves.

recover.sh — The Automated Restore Drill

This is the most important script in this update. A backup you've never tested restoring is a backup you can't trust. recover.sh does exactly what you'd do in a real recovery scenario — it fetches the latest archive from the remote, decrypts it, unpacks it into a staging directory, and reports what came out — but it does it automatically, on a schedule, without requiring you to remember to do it.

#!/bin/bash
# recover.sh — automated restore drill; decrypts and unpacks latest archive to staging

REMOTE_USER="backup"
REMOTE_HOST="your-remote-server.com"
REMOTE_BACKUP_PATH="/backups/pi"
STAGING_DIR="/tmp/pi-backup-drill"
GPG_PASSPHRASE_FILE="/home/pi/.gpg-passphrase"

LOG_FILE="/var/log/pi-backups.log"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] [recover] $1" | tee -a "$LOG_FILE"
}

# Clean up staging dir on exit (success or failure)
cleanup() {
    rm -rf "$STAGING_DIR"
    log "Staging directory cleaned up."
}
trap cleanup EXIT

mkdir -p "$STAGING_DIR"
log "Starting restore drill..."

# Find the most recent archive on the remote
LATEST=$(ssh "${REMOTE_USER}@${REMOTE_HOST}" \
    "find ${REMOTE_BACKUP_PATH} -name '*.gpg' -printf '%T@ %p\n' | sort -n | tail -1 | cut -d' ' -f2-")

if [ -z "$LATEST" ]; then
    log "ERROR: No archives found on remote. Drill failed."
    exit 1
fi

log "Latest archive: ${LATEST}"

# Download, decrypt, and unpack in one pipeline
ssh "${REMOTE_USER}@${REMOTE_HOST}" "cat '${LATEST}'" \
    | gpg --batch --yes \
          --passphrase-file "$GPG_PASSPHRASE_FILE" \
          --decrypt \
    | tar -xzf - -C "$STAGING_DIR" 2>> "$LOG_FILE"

if [ $? -ne 0 ]; then
    log "ERROR: Decrypt/unpack failed. Drill FAILED — check GPG key and archive integrity."
    exit 1
fi

# Report what was restored
FILE_COUNT=$(find "$STAGING_DIR" -type f | wc -l)
TOTAL_SIZE=$(du -sh "$STAGING_DIR" | cut -f1)

log "Drill passed ✓ — ${FILE_COUNT} files, ${TOTAL_SIZE} total. Archive is restorable."
# cleanup runs via trap

The pipeline is the heart of the script: cat streams the archive over the SSH connection, GPG decrypts it on the fly, and tar unpacks it to the staging directory. No temporary file is written to disk during transit — the encrypted bytes stream directly into the decryption process without ever being stored on the Pi in their encrypted form.

The cleanup Trap

The trap cleanup EXIT line ensures the staging directory is always removed when the script exits, whether it succeeds, fails, or is killed with Ctrl+C. Without this, a failed drill would leave a potentially large directory behind in /tmp. The trap pattern is the idiomatic way to handle cleanup in shell scripts — cleaner than duplicating rm -rf at every exit point.

What the Drill Validates

A successful drill confirms four things:

  1. SSH access still works — if key auth broke, the drill fails at the cat step
  2. The archive is intact — a corrupted download or write error would cause GPG or tar to fail
  3. The GPG passphrase is correct — if you rotated the key, the drill catches the mismatch before a real recovery
  4. The archive structure is valid — tar validates the archive format during unpack

Running the drill weekly means you have at most a 7-day window between a backup becoming unrestorable and discovering it. For most self-hosted Pi workloads, that's an acceptable margin.

The Full Cron Schedule

All four scripts now run on a coordinated schedule:

# /etc/crontab — pi-backups full schedule
#
# Nightly backup at 2:00 AM
0  2  * * *  pi  bash -c 'source /home/pi/.pi-backups-env && /home/pi/scripts/backup.sh'

# Healthcheck at 3:00 AM (after backup has had an hour to complete)
0  3  * * *  pi  bash -c 'source /home/pi/.pi-backups-env && /home/pi/scripts/healthcheck.sh'

# Weekly retention pruning, Sunday at 3:30 AM
30 3  * * 0  pi  bash -c 'source /home/pi/.pi-backups-env && /home/pi/scripts/retention.sh'

# Weekly restore drill, Sunday at 4:00 AM (after retention has run)
0  4  * * 0  pi  bash -c 'source /home/pi/.pi-backups-env && /home/pi/scripts/recover.sh'

The Sunday morning window is deliberate. All four scripts run in sequence, spaced apart so each one has finished before the next starts. The retention script runs before the recovery drill so the drill is always testing a fresh archive, not one that's about to be deleted.
