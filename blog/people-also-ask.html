people-also-ask: Mining Google's Question Graph for Content Ideas

Web Development May 2026
TL;DR: people-also-ask is a Node.js CLI tool that takes a seed search query, opens a headless browser via Puppeteer, extracts the "People Also Ask" questions Google surfaces, then recursively expands each question to a configurable depth — producing a deduplicated list of real questions people are searching for. Output is JSON or CSV, ready to feed a content calendar, an FAQ section, or an AI writing prompt.

The Problem with Keyword Research

Traditional keyword research tools give you search volume and competition scores. What they don't give you easily is the actual language people use when asking questions about your topic. That gap matters for content creation: a blog post that directly answers "how do I fix a 502 error on Nginx behind a load balancer?" will outperform one optimised for the generic keyword "Nginx error" because it matches user intent precisely.

Google's "People Also Ask" (PAA) boxes are one of the richest sources of this intent data. They surface real questions, phrased the way actual searchers phrase them, directly inside the search results page. The challenge is that PAA boxes are dynamic — they expand when you click, revealing more questions — and there's no official API to query them programmatically.

That's what people-also-ask solves. It automates the expand-and-collect process using a headless browser, then walks the question graph breadth-first to pull in related questions that most tools miss entirely.

How It Works

Step 1: Seed query → initial PAA box

The tool launches Puppeteer — a headless Chromium instance — and navigates to a Google search results page for your seed query. It waits for the PAA box to render, then extracts all visible question strings from the DOM.

Google's PAA box uses a consistent DOM structure: each question is inside a div[data-q] element (or a closely related selector that has remained stable across recent Google redesigns). The tool reads the data-q attribute rather than the visible text content, which avoids encoding issues with special characters and punctuation.

Step 2: Click each question to reveal more

Each PAA item hides additional questions behind a click. When a user expands a PAA card in a real browser, Google injects 2–4 new questions into the box. The tool replicates this: it iterates through each extracted question element, dispatches a click event, waits for the network request to complete, then re-scrapes the expanded box.

async function expandAndCollect(page, questionElements) {
    const questions = new Set();

    for (const el of questionElements) {
        const text = await el.getAttribute('data-q');
        if (text) questions.add(text);

        await el.click();
        await page.waitForResponse(
            res => res.url().includes('search') && res.status() === 200,
            { timeout: 4000 }
        ).catch(() => {}); // graceful timeout — not every click triggers a request

        const newEls = await page.$$('div[data-q]');
        for (const newEl of newEls) {
            const newText = await newEl.getAttribute('data-q');
            if (newText) questions.add(newText);
        }
    }

    return [...questions];
}

The Set data structure handles deduplication automatically — questions that appear in multiple PAA expansions are collected only once.

Step 3: BFS depth expansion

The most powerful feature is recursive expansion. Each collected question becomes a new seed query for a fresh Google search, and the process repeats. This is breadth-first search through the PAA question graph.

async function mine(seedQuery, options = {}) {
    const { depth = 2, maxQuestions = 50, delay = 1500 } = options;
    const visited = new Set([seedQuery.toLowerCase()]);
    const results = [];

    const queue = [{ query: seedQuery, level: 0 }];

    while (queue.length > 0 && results.length < maxQuestions) {
        const { query, level } = queue.shift();

        const questions = await scrapeQuestions(query);
        results.push(...questions.filter(q => !visited.has(q.toLowerCase())));
        questions.forEach(q => visited.add(q.toLowerCase()));

        if (level < depth) {
            questions.forEach(q => queue.push({ query: q, level: level + 1 }));
        }

        await sleep(delay); // polite crawl rate
    }

    return [...new Set(results)]; // final dedup pass
}

The depth parameter controls how many levels deep the BFS goes. Depth 1 gives you the direct PAA questions for your seed query. Depth 2 follows each of those questions as new queries. Depth 3 follows those as well. At depth 3, a single seed query can surface 40–80 unique questions depending on the topic.

The delay parameter adds a configurable pause between requests. This keeps the crawl rate human-like and reduces the chance of triggering CAPTCHAs.

Installation and Usage

git clone https://github.com/josefresco/people-also-ask.git
cd people-also-ask
npm install

Basic usage — mine questions for a seed query at the default depth of 2:

node index.js "content SEO strategy"

With options:

# Depth 3 expansion, max 60 questions, output as CSV
node index.js "content SEO strategy" --depth 3 --max 60 --format csv --output questions.csv

# JSON output to a file
node index.js "home automation raspberry pi" --format json --output rpi-questions.json

# Increase delay for slower, more polite crawling
node index.js "WordPress plugin development" --delay 2500 --depth 2

CLI flags reference

  • --depth <n> — BFS depth (default: 2, max recommended: 3)
  • --max <n> — Maximum number of unique questions to collect (default: 50)
  • --format <json|csv> — Output format (default: json)
  • --output <file> — Write results to a file (default: stdout)
  • --delay <ms> — Milliseconds to pause between requests (default: 1500)
  • --no-headless — Run Chromium in visible mode for debugging

The Output

JSON output is an array of question strings, ordered roughly by BFS level (seed query's direct questions first, then second-level expansions, and so on):

{
  "query": "content SEO strategy",
  "depth": 2,
  "total": 23,
  "questions": [
    "What is a content SEO strategy?",
    "How do I create an SEO content plan?",
    "What is the difference between SEO and content marketing?",
    "How do you structure a pillar page?",
    "What makes content rank on Google?",
    "How long should a blog post be for SEO?",
    "What is topical authority in SEO?",
    "How do featured snippets work?",
    "..."
  ]
}

CSV format flattens this to one question per row with a level column indicating BFS depth, useful for sorting and filtering in a spreadsheet.

Using the Output for Content Creation

The questions themselves are the deliverable. Here are the three workflows I use most often:

1. FAQ section generation

Feed the question list directly into an FAQ section on an existing page. Each question becomes an <h3> with a concise answer below. FAQs built this way tend to win PAA placements because you're structuring content around the exact phrasing Google already recognises as a question.

2. Content calendar seeding

A single mining run on a broad topic ("home automation") produces enough distinct subtopic questions for 8–12 standalone blog posts. Sort by estimated search intent (informational vs. commercial) and you have a prioritised content calendar with titles that already match what people are searching for.

3. AI prompt enrichment

When drafting content with an AI assistant, prefixing the prompt with "Answer the following questions a reader might have about [topic]:" followed by the mined question list produces dramatically more useful output than a generic "write a blog post about X." The questions act as a structured outline the AI can work through systematically.

Technical Decisions Worth Noting

Puppeteer over fetch

PAA boxes are loaded and expanded via JavaScript — a simple HTTP fetch of a Google results URL gives you the raw HTML before the JS renders, which does not include PAA content reliably. Puppeteer is heavier, but it's the only reliable way to interact with dynamically rendered search results.

Attribute-based selection

Using data-q attribute selectors rather than text content or class names has proven more resilient across Google's ongoing DOM shuffles. Class names on Google search results pages change frequently; semantic data attributes are harder to remove without breaking functionality.

Visited set across levels

The BFS maintains a single visited set shared across all depth levels. This prevents a question encountered at depth 1 from being re-queried at depth 2 when it appears as a result from a different parent question — a common source of duplicate work and API hammering in naive recursive scrapers.

Graceful timeout handling

Not every PAA click triggers a network request — sometimes Google serves the expanded content from cache without a round-trip. The .catch(() => {}) on the waitForResponse call handles this silently rather than throwing, which keeps the loop running without false failures.

Limitations

  • Geographic variation — PAA results are location-sensitive. The tool uses the default locale of the machine running it. For international content research, you'd need to configure Puppeteer's geolocation or use a VPN.
  • CAPTCHA exposure — At high depths or low delays, Google may serve a CAPTCHA. The --no-headless flag lets you complete it manually and resume. There's no automated CAPTCHA solving.
  • PAA availability — Not every query surfaces a PAA box. Navigational queries ("facebook login") and very specific long-tail queries often don't have PAA results. The tool handles this gracefully — it returns an empty array for that query and continues.
  • Snapshot in time — PAA results change based on Google's index and trending topics. A mining run from one week may produce different results than the same run a week later.

Get It

The full source is on GitHub: github.com/josefresco/people-also-ask. It requires Node.js 18+ and will install Chromium automatically via the Puppeteer package on first run.

If you're working with content at scale — running a blog, managing a client's SEO, or building an AI-assisted content pipeline — a PAA mining step before writing is one of the highest-ROI additions you can make to the workflow. The questions are real, the phrasing is exact, and the competitive signal is already baked in: if Google thinks a question belongs next to your topic, ranking for it is worth pursuing.
