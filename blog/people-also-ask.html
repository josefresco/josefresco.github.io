People Also Ask: Building a Bookmarklet to Mine Google's Question Boxes for Content Research

Web Development April 2026
TL;DR: people-also-ask is a vanilla JavaScript bookmarklet you install once from the repo. Run it on any Google search results page and it scrapes every visible "People Also Ask" question from the DOM, deduplicates the list, and surfaces a floating export panel with one-click Copy JSON, Copy CSV, and Copy Text options. No API key, no backend, no browser store review cycle — just a javascript: URI you drag to your bookmarks bar. This post covers the DOM targeting approach, the deduplication strategy, the floating overlay UI pattern, and why a bookmarklet beats a full extension for a tool you use occasionally during research sessions.

Why PAA Questions Are the Best Signal You're Not Using

Google's "People Also Ask" boxes surface the literal questions people are typing into search — not keyword variations a tool infers, but real queries with enough search volume that Google decided they deserved a dedicated accordion UI. For content creators, that's extraordinarily valuable data. Each question is a potential blog post angle, a FAQ item, a section heading, or a long-tail keyword your article should cover.

The problem is that collecting these questions manually is tedious. You open a search, expand a few PAA rows (which dynamically loads more), copy the questions one at a time into a doc or spreadsheet, repeat for every query in your research session. That friction is enough to make most people skip the step entirely.

The tool exists to eliminate that friction. One click, all the questions, in a format you can paste wherever you need them.

The Bookmarklet Deployment Pattern

The first design decision was delivery method: browser extension vs. bookmarklet. A Chrome extension would need a manifest, a store listing, a review cycle, periodic updates to stay compatible with Manifest V3 requirements, and a user who's willing to install something from an unknown publisher. A bookmarklet is a javascript: URI — you get it from the repo README, drag it to your bookmarks bar once, and it runs on any page you invoke it on.

For a tool you use occasionally during research sessions (not constantly in the background), the bookmarklet pattern wins on simplicity. There's no install surface, no permissions prompt beyond what the page already has, and no update to push when Google changes its UI. The user gets a new version by copying a new URI from the repo.

// The installed bookmarklet (minified, lives in your browser)
javascript:(function(){
  const s = document.createElement('script');
  s.src = 'https://raw.githubusercontent.com/josefresco/people-also-ask/main/paa.js?' + Date.now();
  document.head.appendChild(s);
})();

The bookmarklet itself is a two-line loader — it injects the actual script from the repo at runtime, which means the logic can be updated without the user reinstalling anything. The Date.now() cache-busting parameter ensures they always get the current version.

Targeting the PAA DOM Without Brittle Class Names

Google's rendered HTML uses obfuscated, auto-generated class names that change with every UI update. Any scraper that targets .related-question-pair or similar semantic names will break within weeks. The solution is to target structural and content-based attributes instead.

PAA question rows have a few stable characteristics regardless of class name churn:

  • They live inside an element with data-sgrd or similar data attributes marking them as expandable accordion items
  • The question text is always the direct text content of a heading-level element inside each row
  • Each row has an ARIA role of button or contains a [role="button"] descendant for the expand trigger
  • The containing section has visible text "People also ask" somewhere above it in the DOM
function extractQuestions() {
  const questions = new Set(); // deduplicate automatically

  // Target expandable question rows by ARIA role
  const rows = document.querySelectorAll('[role="button"][jsname], [data-ved][jsaction]');

  rows.forEach(row => {
    // Find the first meaningful text node — the question text
    const textEl = row.querySelector('span[jsname], div[data-mb]');
    if (!textEl) return;

    const text = textEl.textContent.trim();

    // Filter: questions end with "?" and are 20–200 chars
    if (text.endsWith('?') && text.length > 20 && text.length < 200) {
      questions.add(text);
    }
  });

  return [...questions];
}

The length filter (> 20 && < 200) catches genuine questions while excluding UI chrome text that accidentally matches the selector. The Set deduplication handles Google's behavior of repeating questions when PAA boxes load dynamically — the same question can appear in multiple PAA clusters on the same results page.

Expanding Hidden Questions

Google's PAA boxes are lazy — they render four questions initially and load more when you expand any of them. The tool handles this in two ways. First, it scrapes what's currently visible. Second, it optionally auto-expands all visible PAA rows before scraping, triggering Google's dynamic loading.

async function expandAndScrape() {
  // Expand all current PAA rows to trigger Google's lazy-load
  const expandables = document.querySelectorAll('[role="button"][jsname]');

  for (const btn of expandables) {
    btn.click();
    // Small delay — Google's JS needs a tick to inject new rows
    await new Promise(r => setTimeout(r, 120));
  }

  // Wait for DOM to settle after the last expansion
  await new Promise(r => setTimeout(r, 400));

  return extractQuestions();
}

The 120ms inter-click delay and 400ms final settle time are empirically tuned. Too fast and Google's mutation-based renderer hasn't had time to inject the new question rows; too slow and the user is waiting noticeably. At these values, a page with three PAA clusters and eight questions each takes about two seconds to fully expand and scrape — fast enough to feel responsive.

The Floating Overlay UI

The result panel injects a positioned overlay into the current page rather than opening a new window or tab. This keeps the context visible — you can see the search results alongside the extracted questions, which is useful when you want to cross-reference.

function renderOverlay(questions) {
  // Remove any previous overlay
  document.getElementById('paa-overlay')?.remove();

  const overlay = document.createElement('div');
  overlay.id = 'paa-overlay';
  overlay.style.cssText = `
    position: fixed;
    top: 20px;
    right: 20px;
    width: 420px;
    max-height: 80vh;
    overflow-y: auto;
    background: #1a1d29;
    border: 1px solid #00d4ff;
    border-radius: 12px;
    padding: 1.25rem;
    z-index: 99999;
    font-family: -apple-system, sans-serif;
    font-size: 13px;
    color: #e2e8f0;
    box-shadow: 0 8px 32px rgba(0,0,0,0.6);
  `;

  overlay.innerHTML = buildOverlayHTML(questions);
  document.body.appendChild(overlay);

  // Wire up export buttons
  overlay.querySelector('#paa-copy-json')
    .addEventListener('click', () => copyToClipboard(JSON.stringify(questions, null, 2)));
  overlay.querySelector('#paa-copy-csv')
    .addEventListener('click', () => copyToClipboard(questions.map(q => `"${q.replace(/"/g, '""')}"`).join('\n')));
  overlay.querySelector('#paa-copy-text')
    .addEventListener('click', () => copyToClipboard(questions.join('\n')));
  overlay.querySelector('#paa-close')
    .addEventListener('click', () => overlay.remove());
}

The overlay is entirely self-contained — inline styles so no external stylesheet is needed, a single removeEventListener-free close button that just removes the element from the DOM, and no globals leaked onto the page. Running the bookmarklet again replaces the existing overlay rather than stacking a second one.

Export Formats

Three export targets cover the downstream use cases:

  • JSON — A structured array. Paste into a file, feed into a script, or use as input to an AI prompt asking it to outline a blog post from the questions.
  • CSV — One question per line, double-quote-escaped. Drop directly into a spreadsheet for a content calendar or keyword tracking sheet.
  • Plain text — A newline-separated list. Paste into a doc, a Notion page, or anywhere plain text belongs. Good for a quick brain-dump without structural overhead.

All three use the browser's navigator.clipboard.writeText() API, which requires the page to be in focus. If the clipboard write fails (clipboard permission denied, which is rare on modern browsers for user-triggered actions), the tool falls back to selecting the text in a temporary <textarea> and calling document.execCommand('copy').

Use Cases

The tool solves a specific workflow: you're researching a topic, you've done several Google searches, and you want to capture all the audience intent signals from those sessions without copying questions by hand.

  • Blog post outlines — Run it on three related queries, copy the questions as JSON, ask an AI to cluster them into section headings. You have an outline in under a minute.
  • FAQ sections — Product pages and support docs benefit from answering the literal questions people ask. PAA questions are a better source than guesswork.
  • Content gap analysis — Compare the PAA questions for your topic against your existing posts. Any question not covered is a content gap.
  • Long-tail keyword research — PAA questions often contain natural-language phrasing that matches voice search queries — phrases that traditional keyword tools undercount.

Limitations

The tool is honest about what it can't do. Google's DOM is not a stable API — the selector approach that works today will eventually need updating as Google ships UI changes. The repo includes a simple test page for validating the selectors without running live against Google. When the selectors break, the fix is typically a one-line update to the query string.

The tool also doesn't paginate. It extracts questions from the current SERP only. If you need deeper coverage — questions from 10 related queries, say — you run it on each search manually. Automating multi-query scraping would require a headless browser and crosses into territory that violates Google's Terms of Service.

Finally, the auto-expand feature has a ceiling. Google limits how many PAA rows it will lazy-load in a single page session. In practice you get around 15–20 unique questions per query before the "See more" button stops producing new rows — which is usually more than enough for a content research session.

The Repo

The source is at github.com/josefresco/people-also-ask. The README has the bookmarklet URI, installation instructions, and the test page for validating selectors after Google UI updates. The entire implementation is a single ~150-line JavaScript file — small enough to read and audit in a few minutes before trusting it on your browser.
