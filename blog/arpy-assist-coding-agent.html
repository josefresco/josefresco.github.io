arpy-assist: From Qwen CLI Frontend to Raspberry Pi Coding Agent

Raspberry Pi March 2026
TL;DR: arpy-assist started as a web UI wrapping Qwen CLI on a Raspberry Pi. It's since grown into a local AI coding agent — one that detects code blocks in Qwen's responses, offers to execute them directly on the Pi via the raspberry-pi-agent dispatcher, and streams back the result. No cloud APIs, no data leaving the network, no copy-pasting between tabs.

The Gap in the Original Design

The first version of arpy-assist solved a real annoyance: accessing Qwen CLI from a phone or another computer without SSH. You'd open the web interface, type a prompt, and get a response streamed back from the Pi. Useful for asking questions, generating scripts, talking through a problem.

But every time Qwen produced a code block, the workflow broke down. You'd read the code in the browser, then open a new tab to SSH into the Pi, paste it into a file, make it executable, run it, and come back to check the output. Five steps for what should be one.

The coding agent capability exists to close that gap. When arpy-assist sees a code block in Qwen's response, it wraps it in a "Run on Pi" button. One click sends that code to the raspberry-pi-agent dispatcher, which executes it and streams the output back into the same conversation thread.

How Code Detection Works

Qwen's output is Markdown. Code blocks come back as standard fenced blocks — triple backticks with an optional language tag. The response parser in arpy-assist walks the rendered output and wraps each detected code block with an execution affordance:

function wrapCodeBlocks(responseText) {
    const fencePattern = /```(\w+)?\n([\s\S]*?)```/g;
    return responseText.replace(fencePattern, (match, lang, code) => {
        const language = lang || 'bash';
        const escaped = encodeURIComponent(code.trim());
        return `
<div class="code-block-wrapper">
  <pre><code class="language-${language}">${code.trim()}</code></pre>
  <button class="run-on-pi"
          data-code="${escaped}"
          data-lang="${language}">
    ▶ Run on Pi
  </button>
</div>`;
    });
}

The language tag matters: bash and shell blocks execute directly via the dispatcher's shell runner. python blocks are written to a temp file and run with the system Python. javascript blocks are run with Node.js if it's available. Any other language shows the button as disabled with a "Not supported" tooltip rather than silently failing.

The Execution Pipeline

When the "Run on Pi" button is clicked, arpy-assist sends a POST to its own backend:

POST /api/execute
{
  "code": "import psutil\nprint(psutil.cpu_percent(interval=1))",
  "language": "python",
  "timeout": 30
}

The arpy-assist backend is the same Python Flask server that handles the Qwen chat API. It forwards the execution request to the raspberry-pi-agent dispatcher over its local socket interface — the same dispatcher used by the Telegram bot and the backup scripts.

The dispatcher handles the actual subprocess management: spawning the process, capturing stdout and stderr, enforcing the timeout, and cleaning up temp files. arpy-assist gets the result back as a JSON payload and appends it to the conversation thread as a collapsible output block:

{
  "exit_code": 0,
  "stdout": "14.2\n",
  "stderr": "",
  "duration_ms": 1043,
  "truncated": false
}

Non-zero exit codes render the output block with a red border and the stderr content expanded by default. Timeouts return a synthetic error with the partial stdout captured up to the cutoff.

Dry-Run Mode and Confirmation

Not every code block should execute without a second look. Qwen sometimes produces code that's directionally correct but makes assumptions about file paths or environment variables that don't hold on my Pi setup. Running it blindly would create files in the wrong place or fail halfway through with a confusing error.

Dry-run mode is always on by default. When you click "Run on Pi," arpy-assist first renders a confirmation dialog that shows:

  • The exact command or script that will run
  • The working directory (defaults to /home/pi unless overridden)
  • The configured timeout
  • A warning if the code contains potentially destructive patterns (rm -rf, DROP TABLE, redirects to system paths)

You confirm or cancel. If you confirm, execution proceeds. If you cancel, the code block stays visible and you can edit it inline before retrying. There's a session-level setting to disable the confirmation for a session if you're in an exploratory flow and the constant prompts become friction — but it resets on every page load.

Inline Editing Before Execution

The other pattern I kept running into: Qwen's code is right in structure but needs a variable swapped out — a hardcoded path, a placeholder API key, a hostname that matches the model's training data rather than my actual Pi. Copying to a text editor, editing, pasting back, clicking run is four steps. The inline editor cuts it to one.

Each code block in the interface is also a contenteditable pre-code block. Clicking the code switches it from display mode to edit mode. The "Run on Pi" button reads the current content of the block at click time, not the original Qwen output — so edits are automatically picked up without any extra save step.

runButton.addEventListener('click', () => {
    // Read current content from the editable block,
    // not the original data-code attribute
    const currentCode = codeBlock.innerText.trim();
    executeOnPi(currentCode, language);
});

Context Injection: Giving Qwen Pi-Specific Awareness

One limitation of running a local model like Qwen on a Raspberry Pi is that it has no knowledge of your specific environment. It doesn't know your Pi's hostname, what services are running, what Python packages are installed, or how your directory structure is organized. Left to its own devices, it generates generic code that often needs manual adaptation.

arpy-assist addresses this with a system context prompt that's prepended to every conversation. The context is generated at session start by querying the raspberry-pi-agent for a lightweight environment snapshot:

SYSTEM CONTEXT (injected at session start):
You are a coding assistant running on a Raspberry Pi 5.
Environment:
- Hostname: arpy
- OS: Raspberry Pi OS Bookworm (64-bit)
- Python: 3.11.2 (/usr/bin/python3)
- Node.js: 20.11.0 (/usr/local/bin/node)
- Available services: mosquitto, nginx, jf-website-monitor, arpy-assist
- Home directory: /home/pi
- App data directory: /opt/apps
- Installed Python packages: psutil, flask, requests, paho-mqtt, schedule

When generating code, use these paths and package names.
Prefer scripts that can be run without root unless root is genuinely required.

This context is refreshed on each new conversation (not on each message — that would add overhead). The result is code that references real hostnames, uses the actual Python path, and imports packages that are actually installed rather than hallucinating pip install steps for packages already present.

Running Qwen on Pi 5 Hardware

The previous post covered the setup: Qwen running via Ollama on a Raspberry Pi 5 with 8GB RAM. A few things worth adding now that the agent has been in use for a while:

Model Size vs. Code Quality Tradeoff

I run qwen2.5-coder:7b by default. The 1.5b model is noticeably faster (responses in ~8 seconds vs. ~25 seconds for 7b) but produces code that more frequently needs correction for anything beyond simple shell commands. For coding tasks specifically — which is most of what arpy-assist is used for — the 7b model's accuracy improvement is worth the latency.

Thermal Throttling

Extended coding sessions push the Pi 5's CPU to sustained load. Without active cooling, the SoC throttles at around 80°C, which extends response times noticeably. I'm running a Geekworm case with active fan cooling — it keeps the CPU under 65°C under sustained load and eliminates throttling entirely. If you're running Qwen on a Pi and seeing inconsistent response times, thermal throttling is the first thing to check with vcgencmd measure_temp.

Concurrent Requests

Ollama queues concurrent requests rather than parallelizing them. If two devices send prompts at the same time, the second one waits. For a personal tool this is fine — there's only ever one person using it. The queue does mean that if you accidentally double-submit a long prompt, you'll see both responses eventually rather than one being dropped.

The Agent Loop in Practice

A typical session for writing a new automation script on the Pi now looks like this:

  1. Open arpy-assist on my phone while the Pi is running a job I want to script
  2. Describe what I want: "Write a Python script that checks disk usage on /opt/apps and sends a Telegram message if any subdirectory exceeds 500MB"
  3. Qwen returns the script. I scan it — the paths look right from the injected context, the Telegram bot token is a placeholder I need to fill in
  4. Click the code block, replace the placeholder with the real token inline
  5. Click "Run on Pi," confirm in the dialog, watch the output stream back
  6. If the script needs tweaking, I either edit it inline and re-run, or ask Qwen a follow-up ("the Telegram message is being sent but the directory names are truncated — fix that")

The whole loop from idea to running script is usually under 3 minutes, compared to 10–15 minutes when the workflow involved SSH, a text editor, and manual copy-paste.

What I Got Wrong

Trusting Code Output Too Early

The first version of the execution feature had no confirmation dialog — just a direct "Run" button. I broke a monitoring service by running a Qwen-generated systemd reload command that had the wrong service name. The service it reloaded happened to conflict with another service's port binding. Adding the dry-run preview and the destructive-pattern warnings came directly from that incident.

Streaming vs. Complete Responses

Qwen streams responses token by token. The original code-block detection ran on the complete response after streaming finished, which meant you'd watch the code appear token by token and then wait for the "Run on Pi" button to appear after a noticeable pause. Switching to incremental parsing — detecting the closing triple-backtick fence as it streams — makes the button appear as soon as the code block is complete, before the rest of the response has finished rendering.

Session Context Accumulation

Qwen's context window is finite. Long coding sessions where I kept iterating on the same script would eventually hit the context limit, causing Qwen to start forgetting earlier parts of the conversation. The fix was a "Summarize and continue" button that sends the conversation history to Qwen with the prompt "Summarize this conversation in 3 bullet points, focusing on the code that was written and the current state of the script." The summary replaces the full history in the context, keeping the conversation focused without losing the important state.

What's Next

The piece I want to add next is file system awareness — the ability to tell arpy-assist "edit /opt/apps/monitor/config.json" and have it read the current file content, pass it to Qwen as context, apply the suggested changes, and write the result back. Right now that workflow still requires SSH for the file read/write steps. The raspberry-pi-agent dispatcher already has file read and write commands; it's a matter of wiring them into the arpy-assist agent loop.

The other piece is conversation persistence — saving session history to a local SQLite database so you can pick up a coding thread where you left it rather than starting fresh every time. Useful for multi-session projects where you're iterating on the same script over several days.

Conclusion

arpy-assist started as a convenience wrapper around Qwen CLI. Adding code execution capability turned it into something more genuinely useful: a local AI coding assistant that can take a natural language description of an automation task and go from prompt to running script in a single session, on a $90 piece of hardware, without any data leaving the network.

The pieces that made it work weren't novel — code block detection, a dispatcher for subprocess execution, a confirmation dialog for safety. But assembling them on a constrained device with a local model changed the character of the tool from "chat with AI" to "get things done with AI."
