Building a Gamified Time Tracking Dashboard with the Breeze PM API

Web Development August 2025
TL;DR: I built a gamified time tracking dashboard on top of the Breeze PM API, deployed on Netlify with serverless functions to keep API credentials off the client. It turns billable hours into a level progression system — you earn XP, unlock titles, and see yourself on a leaderboard. Whether that's motivating or anxiety-inducing depends entirely on your team culture.

The Problem with Standard Time Tracking UIs

Every project management tool has a time tracking view. They all look roughly the same: a table of entries, a total at the bottom, maybe a bar chart if you're lucky. They're functional, but they communicate nothing about progress or trajectory. Is 22 hours this week good? Are you on pace for your targets? How does that compare to last month?

The raw numbers tell you what happened. They don't tell you how to feel about it, which is what drives behavior. I wanted to build something that added that layer — that translated tracked hours into a narrative about progress. Gamification is an obvious mechanism for that.

Breeze PM is the project management tool I use for client work. It has a well-documented REST API that exposes time entries, projects, and user data. That was the data source. The question was what to build on top of it.

The Breeze PM API

Breeze exposes a REST API with token authentication. The endpoints I used:

  • GET /projects — list all active projects with metadata
  • GET /projects/:id/time_entries — time entries for a specific project, filterable by date range and user
  • GET /users — team members with IDs for associating entries to people

The API doesn't have a dedicated "all time entries across all projects" endpoint, so the dashboard fetches entries per project and aggregates them client-side. This works fine for teams with under ~50 active projects. For larger installations, you'd want to cache the aggregation server-side rather than re-fetching on every page load.

Serverless Functions: Keeping Credentials Off the Client

The Breeze API token gives full read (and write) access to your workspace. Putting it in client-side JavaScript would expose it to anyone who opened DevTools. The solution is a thin serverless proxy — a Netlify Function that holds the token, receives requests from the frontend, calls the Breeze API, and returns the response.

// netlify/functions/breeze-proxy.js
const fetch = require('node-fetch');

exports.handler = async (event) => {
    const { endpoint } = JSON.parse(event.body || '{}');

    // Allowlist — only permit fetching from known safe endpoints
    const allowed = ['/projects', '/users'];
    const isAllowed = allowed.some(e => endpoint.startsWith(e));

    if (!isAllowed) {
        return { statusCode: 403, body: JSON.stringify({ error: 'Forbidden' }) };
    }

    const response = await fetch(
        `https://api.breeze.pm/api/v1${endpoint}`,
        {
            headers: {
                'Authorization': `Bearer ${process.env.BREEZE_API_TOKEN}`,
                'Content-Type': 'application/json'
            }
        }
    );

    const data = await response.json();
    return {
        statusCode: 200,
        body: JSON.stringify(data)
    };
};

The allowlist on endpoint is important. Without it, the proxy becomes a general-purpose credential relay — any caller could pass any Breeze API path and read data they're not supposed to. The allowlist restricts it to the specific endpoints the dashboard needs.

The API token lives in a Netlify environment variable, set through the Netlify dashboard, never committed to the repo.

The Gamification Layer

The core idea: tracked hours earn XP, XP unlocks levels, levels have titles. The progression curve is tuned so that consistent tracking feels rewarding without making a slow week feel catastrophic.

XP and Level Calculation

// XP is earnings-based (hourly rate × hours tracked this month)
function calculateXP(entries, hourlyRate) {
    const totalHours = entries.reduce((sum, e) => sum + e.minutes / 60, 0);
    return Math.floor(totalHours * hourlyRate);
}

// Level thresholds use a square-root curve so early levels feel fast
// and later levels require sustained effort
function getLevel(xp) {
    return Math.floor(Math.sqrt(xp / 100));
}

const LEVEL_TITLES = [
    'Apprentice', 'Journeyman', 'Craftsman', 'Expert',
    'Senior', 'Lead', 'Principal', 'Architect',
    'Director', 'Grandmaster'
];

function getLevelTitle(level) {
    const index = Math.min(level, LEVEL_TITLES.length - 1);
    return LEVEL_TITLES[index];
}

The square-root curve means the gap between level 1 and level 2 is small, but the gap between level 8 and level 9 is substantial. This matches how gamified systems typically feel motivating early on and then reward consistency over time rather than just one big push.

The Leaderboard

The leaderboard ranks team members by XP for the current month. It's the most contentious feature — I'll come back to why — but technically it's straightforward: fetch entries for all users, calculate XP per user, sort descending, render a ranked list with their level title and a progress bar to the next level.

function buildLeaderboard(userEntries, hourlyRates) {
    return Object.entries(userEntries)
        .map(([userId, entries]) => {
            const xp = calculateXP(entries, hourlyRates[userId] || DEFAULT_RATE);
            const level = getLevel(xp);
            const nextLevelXP = Math.pow(level + 1, 2) * 100;
            const progress = (xp - Math.pow(level, 2) * 100) / (nextLevelXP - Math.pow(level, 2) * 100);
            return { userId, xp, level, title: getLevelTitle(level), progress };
        })
        .sort((a, b) => b.xp - a.xp);
}

Project Progress Rings

Each active project shows a circular progress ring representing hours tracked vs. the project's estimated hours. This is rendered as an SVG with a stroke-dashoffset animation — the ring fills in as hours accumulate. Projects over 80% of their estimate turn amber; projects over 100% turn red.

function renderProgressRing(tracked, estimated, element) {
    const radius = 36;
    const circumference = 2 * Math.PI * radius;
    const ratio = Math.min(tracked / estimated, 1);
    const offset = circumference * (1 - ratio);

    const color = ratio >= 1 ? '#ef4444'
                : ratio >= 0.8 ? '#f59e0b'
                : '#00d4ff';

    element.style.strokeDasharray = circumference;
    element.style.strokeDashoffset = offset;
    element.style.stroke = color;
    element.style.transition = 'stroke-dashoffset 0.8s ease, stroke 0.3s ease';
}

Real-Time Updates

The dashboard polls the Breeze API via the serverless proxy every 5 minutes and re-renders changed values with a brief highlight animation. No WebSockets needed — time tracking data doesn't change fast enough to justify persistent connections. The 5-minute interval is a tunable constant at the top of the main JS file.

On the first load, all data fetches happen in parallel using Promise.all to minimize time-to-interactive:

const [projects, users] = await Promise.all([
    breezeProxy('/projects'),
    breezeProxy('/users')
]);

// Then fetch time entries for all projects in parallel
const entriesByProject = await Promise.all(
    projects.map(p => breezeProxy(`/projects/${p.id}/time_entries?from=${monthStart}`))
);

What Works, What Doesn't

What Works

  • Daily tracking habit — Seeing the XP counter and knowing a log entry will push you closer to the next level is a small but real nudge toward consistent tracking. Before this dashboard, I'd sometimes forget to log time until the end of the week.
  • Project health at a glance — The progress rings make it immediately obvious which projects are running hot before they hit 100%. That's the genuinely useful business insight buried in the gamification.
  • Zero backend to maintain — Netlify Functions handle the API proxy. There's no server to update, no database to back up, no process to restart. It just works.

What Doesn't Work

  • Leaderboards in shared team contexts — I built this for my own use as a personal motivation tool. If a manager uses it to compare team members' output, it creates the wrong incentives: people start optimizing for tracked hours rather than effective work. One team lead who saw a demo immediately asked about using it to "identify who's pulling their weight." That's exactly the wrong use case.
  • Hourly rate assumptions — XP is earnings-based, which means team members at different rates earn XP at different rates even if they work the same hours. That's fine for personal use where you control the rate. It makes the leaderboard misleading in a team context.
  • No historical trend — The current month resets to zero on the first. There's no month-over-month view showing whether this month is better than last month. That context is often more motivating than absolute numbers.

Lessons Learned

The technical parts of this project were straightforward. The interesting design challenge was understanding when gamification helps and when it doesn't.

Gamification works well when the metric it's measuring is tightly correlated with the outcome you actually want. Hours tracked is a reasonable proxy for productive work if you're a freelancer billing hourly — but it's a poor proxy for actual output quality, client satisfaction, or creative work. Building a system that rewards the proxy over the real thing can subtly push behavior in the wrong direction.

For personal use, as a way to build a consistent logging habit and stay aware of project health, it works. I've been using it daily since I built it. For team deployment, I'd strip out the leaderboard and replace it with individual trend views — same data, less competitive framing.

Running It Yourself

The repo is on GitHub. You'll need a Breeze PM account with API access enabled, a Netlify account for the serverless function deployment, and your API token added as a BREEZE_API_TOKEN environment variable in Netlify's dashboard.

# Clone and deploy
git clone https://github.com/josefresco/breeze-time-dash.git
cd breeze-time-dash
netlify deploy --prod

The config.js file at the root has tunable constants for hourly rate, polling interval, level titles, and XP thresholds.
