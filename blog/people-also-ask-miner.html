People Also Ask Miner: A Zero-Dependency Node.js Tool for Content Research

Content Tools May 2026
TL;DR: people-also-ask is a lightweight Node.js server that accepts a search keyphrase, queries SerpApi, and returns up to 10 "People Also Ask" questions as a JSON array. It ships with a clean browser UI and one-click copy — and it has zero npm dependencies. The whole stack is four files: a server, a CORS proxy, an HTML frontend, and a Windows batch launcher.

Why PAA Questions Matter for Content

The "People Also Ask" box that appears in Google search results is one of the more useful signals in SEO — not because it tells you what to rank for, but because it tells you what questions real users are already asking around a topic. Those questions are the raw material for FAQ sections, blog subheadings, video scripts, and structured Q&A markup that search engines reward.

The problem with PAA questions is that gathering them manually is tedious. You type a search, screenshot the PAA box, copy each question by hand, and repeat for every keyphrase you're researching. It takes five minutes per query when you need it to take five seconds.

I needed a way to pipe keyphrases in and get questions out — fast, with clipboard copy, without installing a browser extension or signing up for a SaaS dashboard. So I built one.

The Tool: Four Files, Zero Dependencies

The repo is intentionally minimal:

people-also-ask/
├── paa-server.js      # HTTP server on port 3131
├── paa-proxy.js       # CORS proxy for standalone use
├── paa-miner.html     # Browser frontend
└── paa-miner.bat      # Windows one-click launcher

There are no node_modules, no package.json dependencies, no bundler. The server uses Node's built-in http and https modules, which have been part of Node's standard library since v0.x. If you have Node.js v14 or later installed, you can run this with a single node paa-server.js and it works.

How the Server Works

The server exposes one endpoint:

GET /search?q=<keyphrase>

When a request comes in, the server constructs a SerpApi URL with the search query and your API key, fetches the response over HTTPS, and plucks the related_questions array from the JSON payload. It caps the result at 10 questions and returns them as a JSON array of question strings:

["What are the best free tools for content research?",
 "How do bloggers find SEO keywords for free?",
 "What is the People Also Ask box on Google?",
 "How do I use SerpApi for content marketing?",
 ...]

The implementation uses Node's native https.get() rather than fetch or an HTTP library, which is the key to keeping the dependency count at zero:

const https = require('https');
const http = require('http');
const url = require('url');

const SERPAPI_KEY = 'YOUR_SERPAPI_KEY_HERE';
const PORT = 3131;

const server = http.createServer((req, res) => {
  const parsed = url.parse(req.url, true);

  if (parsed.pathname !== '/search') {
    res.writeHead(404);
    return res.end('Not found');
  }

  const q = parsed.query.q || '';
  if (!q) {
    res.writeHead(400);
    return res.end(JSON.stringify({ error: 'Missing query parameter: q' }));
  }

  const apiUrl = `https://serpapi.com/search.json?q=${encodeURIComponent(q)}&api_key=${SERPAPI_KEY}&engine=google`;

  https.get(apiUrl, (apiRes) => {
    let data = '';
    apiRes.on('data', chunk => { data += chunk; });
    apiRes.on('end', () => {
      const json = JSON.parse(data);
      const questions = (json.related_questions || [])
        .slice(0, 10)
        .map(q => q.question);

      res.writeHead(200, { 'Content-Type': 'application/json', 'Access-Control-Allow-Origin': '*' });
      res.end(JSON.stringify(questions));
    });
  }).on('error', (err) => {
    res.writeHead(500);
    res.end(JSON.stringify({ error: err.message }));
  });
});

server.listen(PORT, () => console.log(`PAA server running at http://localhost:${PORT}`));

The Access-Control-Allow-Origin: * header means the API can be called directly from the browser frontend without a proxy — though the proxy is there for environments where that's needed.

The CORS Proxy

paa-proxy.js is an alternative entry point for environments where you want to serve the frontend HTML directly from the Node process rather than via a separate file server. It handles the same /search endpoint but also serves paa-miner.html at the root path, which means you can point your browser at http://localhost:3131/ and get both the UI and the API from one process.

The proxy approach is useful on Windows where running two command-line processes and keeping track of ports is friction the tool doesn't need to add. One command, one URL, done.

The Browser Frontend

paa-miner.html is a single self-contained HTML file with no external dependencies. It has an input field for the keyphrase, a submit button that calls /search?q=, and a results list that renders each question. The two pieces of UI that matter most are:

Per-question copy buttons. Each result renders with a clipboard icon button. Clicking it copies just that question to the clipboard — useful when you want to pull one question into a doc without selecting and copying the text manually.

Copy All button. When you've got 8–10 questions back and want to paste them all into a spreadsheet or document, Copy All copies the full list as newline-separated text. One click, everything on the clipboard.

The frontend's fetch call is straightforward:

async function search(query) {
  const res = await fetch(`/search?q=${encodeURIComponent(query)}`);
  const questions = await res.json();
  renderResults(questions);
}

function renderResults(questions) {
  const list = document.getElementById('results');
  list.innerHTML = '';
  questions.forEach(q => {
    const li = document.createElement('li');
    li.innerHTML = `
      <span class="question-text">${q}</span>
      <button onclick="copyToClipboard('${q.replace(/'/g, "\\'")}')">⎘</button>
    `;
    list.appendChild(li);
  });
}

function copyToClipboard(text) {
  navigator.clipboard.writeText(text);
}

function copyAll() {
  const questions = [...document.querySelectorAll('.question-text')]
    .map(el => el.textContent)
    .join('\n');
  navigator.clipboard.writeText(questions);
}

The Windows Launcher

paa-miner.bat is a batch file for Windows users who don't want to open a terminal, type a command, and then separately open a browser. Double-clicking the .bat file starts the Node server and opens the browser to http://localhost:3131/ in one shot:

@echo off
start "" http://localhost:3131/
node paa-proxy.js
pause

The start command fires the browser open immediately, then Node starts. In practice the server is ready within a few hundred milliseconds, well before the browser finishes loading, so there's no race condition in normal use. The pause at the end keeps the terminal window open so you can see any error output if something goes wrong.

SerpApi vs. Direct Scraping

The tool uses SerpApi rather than scraping Google directly. That's a deliberate choice, not a limitation.

Scraping Google's HTML directly is fragile — Google changes its markup without notice, CAPTCHAs appear under load, and IP-based rate limiting kicks in quickly when you're running searches programmatically. SerpApi handles all of that: it maintains parsers for Google's current markup, provides structured JSON responses, and takes responsibility for the scraping infrastructure. The free tier includes 100 searches per month, which is more than enough for periodic content research sessions.

The tradeoff is that you need an API key and a SerpApi account. For a personal content research tool that runs locally, that's an acceptable dependency — it's a credential, not a library.

Practical Workflow

The typical session looks like this: I open paa-miner.bat (or node paa-proxy.js on Mac/Linux), type the topic I'm writing about, and get back 8–10 questions in about a second. I scan the list for questions that match what I want to cover, hit Copy All, and paste into a doc. Then I type the next keyphrase.

Where this pays off most is FAQ section planning. A well-structured FAQ with 5–8 questions per page, each drawn from actual PAA results, tends to pick up featured snippet placement for the questions it answers. The tool removes the manual friction that made that process feel like more work than it was worth.

Setup in Three Steps

  1. Clone the repo: git clone https://github.com/josefresco/people-also-ask.git
  2. Open paa-server.js and replace YOUR_SERPAPI_KEY_HERE with your SerpApi key
  3. Run node paa-proxy.js (or double-click paa-miner.bat on Windows) and open http://localhost:3131

No install step. No build step. The tool is ready to use in the time it takes to paste an API key.

What I Might Add Next

A few things that would make this more useful without compromising the zero-dependency principle:

  • Batch input: Accept a newline-separated list of keyphrases and process them sequentially, outputting all results to a single downloadable text file. Useful for pre-research sessions before starting a content calendar.
  • Local result caching: Write results to a JSON file keyed by query, so repeated lookups for the same term don't consume API quota. Node's built-in fs module handles this without adding a dependency.
  • Category grouping: SerpApi's response includes context for some PAA questions. Grouping results by intent cluster (how-to, what-is, comparison) could make the list easier to scan.

For now the tool does what it needs to: take a keyphrase, return questions, copy to clipboard. That's the whole job, and keeping it that focused is most of the reason the implementation stayed this simple.

The Zero-Dependency Constraint

The no-npm-dependencies rule wasn't a design principle when I started — it emerged from wanting the tool to be immediately runnable. No npm install means no lockfile drift, no vulnerability alerts for transitive dependencies you didn't choose, and no moment where someone clones the repo and hits a Node version mismatch in a dependency.

Node's built-in http, https, url, and fs modules cover everything this tool needs. They're stable, well-documented, and available anywhere Node runs. For tools that are primarily configuration wrappers around an external API, that's often enough.
